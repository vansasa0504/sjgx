## 2. 第一性原理分析

### 2.1 用户真正要解决的问题

P0-P2 的代码、自动化测试、脚本和审查已形成开发闭环，但原始招采要求中的许多结论必须依赖**目标环境、真实拓扑、真实数据规模、机构权限和第三方测评**。当前问题不是继续堆叠功能，而是发布决策者无法仅靠现有材料安全、诚实地回答：

1. 哪些需求已有可复核的开发证据？
2. 哪些只是在开发环境验证过，尚不能推导生产达标？
3. 哪些是功能尚未交付、外部规范未提供或必须由机构签署的事项？
4. 在何种严格条件下可以准入、拒绝准入或进行书面限期豁免？

若不建立统一的状态、证据和决策规则，就可能把脚本、Mock、单测、文档或代码审查错误地当作生产验收结果，造成虚假准入风险。

### 2.2 最小可行结果

不执行生产操作的最小结果是一个版本化、可独立审查的**上线验收准备包**：

1. 46 条 FR 和九大模块的证据索引；
2. 覆盖功能、数据库、持久化、安全、测试、性能、可用性、兼容性、合规、运维、文档、发布回退的门禁矩阵；
3. 针对未实测 NFR 的生产等价补测计划和结果回填规则；
4. 把“待验证”“未实现”“外部依赖”和“可申请豁免”严格区分的差距台账；
5. 发布准入、风险豁免、上线观察和回退清单；
6. 文档、培训、SLA、值守与巡检的非代码证据清单。

最小结果**不等于**正式上线批准，更不等于完成性能、容灾、安全或等保验收。它只使后续有权角色能够基于相同事实作出可追溯决定。

### 2.3 核心输入

| 输入类别 | 必要输入 | 来源/约束 |
|---|---|---|
| 需求基线 | FR-101~FR-906（46 条）、NFR-P01~NFR-G01（24 个唯一编号，按复合阈值拆分为 27 条原子门禁）和 P3 RLS/NFR-R | `docs/requirements.md`；不得改写原阈值 |
| 开发证据 | commit、测试、审查、阶段报告、脚本、交付文档 | 仅引用版本化文件和已存在证据；不补造输出 |
| 生产待验证证据 | 压测/稳定性/演练/扫描/兼容性原始结果 | P3 仅定义模板与受控位置；待授权环境执行 |
| 外部条件 | 机构接口规范、目标环境、容量、变更窗口、审批角色 | 未提供时为 `BLOCKED`，不是 `PASS` |
| 风险决策 | 豁免、责任、到期日、补齐任务、回退条件 | 必须由有权业务/安全/运维角色确认；不得由开发方代签 |

### 2.4 核心输出

| 输出 | 最低内容 | 不可接受的替代物 |
|---|---|---|
| FR 证据索引 | FR 分组、状态、证据路径、环境、责任角色、下一动作 | 仅写“已实现”或仅指向源代码 |
| 发布门禁矩阵 | 门禁 ID、来源、阈值/规则、最低证据、环境、状态、阻断级、责任角色、到期日 | 把开发单测/脚本检查直接标生产 PASS |
| 补测计划 | 前置、步骤、指标、起止时间、PASS/FAIL、停止条件、清理/回退、受控归档位置 | 只罗列命令、不写判定和环境 |
| 差距台账 | 分类、风险、阻断结论、责任方、证据/豁免需求 | 把未实现功能表述为“待测试” |
| 准入清单 | 准入决策、残留项、角色占位、豁免字段、观察/回退触发条件 | 口头豁免、无到期日的风险接受 |
| 服务交付清单 | 文档/培训/SLA/值守/巡检材料、版本、责任角色、受控位置 | 把合同/组织承诺表述为代码已验证 |

### 2.5 不可省略的处理过程

1. **冻结基线**：以当前 `master` 的 P0-P2 产物和原始需求为唯一事实源，记录本阶段的 commit 基线；不从记忆推断结果。
2. **逐项取证**：对 46 FR、24 个唯一 NFR（拆分后的 27 条原子门禁）和流程门禁逐一寻找已有证据；记录证据来源、环境、执行日期、采集角色与证据层级（开发/生产等价/生产）。
3. **归类而非粉饰**：证据不足、环境不匹配、外部规范缺失或功能未实现时，标为 `BLOCKED`/`FAIL`，不得升格为 `PASS`。
4. **定义可判定补测**：对能在获授权环境补测的项，编写前置、输入、步骤、时间边界、指标、原始输出位置、PASS/FAIL 和停止/回退条件。
5. **识别不可由测试解决的缺口**：MFA/IAM/SSO、字段级 ABAC、证书认证、真实财务/采购/监管对接、等保测评、服务合同等应归入功能/外部/组织依赖，不伪装为测试任务。
6. **汇总准入规则**：矩阵中存在任一 `FAIL` 或未批准的阻断 `BLOCKED` 时，结论必须为“不准入”；仅完整、限期且有授权的例外可为 `WAIVED`。
7. **自校验**：验证所有仓库相对链接存在，门禁状态在枚举内，FR/NFR 覆盖数匹配，文档不含敏感信息；审查 Git diff 仅含本阶段允许的文档。

### 2.6 核心能力与增强能力区分

| 类型 | 能力 | 是否本次必须实现 | 说明 |
|---|---|---:|---|
| 核心 | FR 证据索引 | 是 | 46 FR 均可追溯，含外部依赖与缺口 |
| 核心 | 上线门禁矩阵与状态机 | 是 | 全部流程门禁；`PASS/FAIL/BLOCKED/WAIVED` 严格定义 |
| 核心 | 生产等价补测计划 | 是 | 覆盖 P/A/S/E/C/M/U/G 全部未实测项；其中 E01~E03 以独立 PV-ARCH 卡覆盖，定义结果回填 |
| 核心 | 安全/合规/外部依赖台账 | 是 | 真实区分待验证与未实现/未提供 |
| 核心 | 准入、豁免、观察与回退清单 | 是 | 决策不能靠口头判断 |
| 核心 | 服务交付证据清单 | 是 | 文档、培训、SLA、值守、巡检分层 |
| 增强 | 自动采集门禁证据的工具/CI 看板 | 否 | 未有稳定目标环境与受控证据系统，避免引入工具复杂度 |
| 增强 | 执行真实压测/DAST/容灾切换/发布 | 否 | 需机构环境、书面授权和变更窗口，属于后续实施 |
| 增强 | 实现 MFA/IAM/SSO/ABAC/证书认证或真实外部对接 | 否 | 是独立功能/外部联调任务，不能在文档任务中混入 |

### 2.7 最小改动路径

1. 保留现有 `docs/requirements.md` 与 `tasks/requirement-analysis.md` 的原 FR/NFR；仅追加 P3 附录，不重写既有基线。
2. 新建 `delivery/release-readiness/` 作为 P3 验收材料的唯一目录，避免散落修改既有 P0-P2 报告。
3. 在该目录以 Markdown 生成七份相互引用的最小文档：README、证据索引、门禁矩阵、补测计划、差距台账、准入清单、交付服务清单。
4. 引用已有 `perf/`、`security/`、`delivery/chaos-drill/`、`delivery/backup-restore/` 的脚本和报告，不复制、不改造、不执行。
5. 用 Markdown 表格和受限状态枚举完成可读、可审计的交付；不引入数据库、服务、前端、依赖或自动化平台。

### 2.8 避免过度设计的约束

- 只定义证据结构和执行计划，不实现新的发布平台、工作流引擎、风险系统或 CI 插件。
- 不把每个单测复制进矩阵；一组 FR/NFR 可引用同一审查/测试证据，但必须清楚说明覆盖边界。
- 不将没有真实环境的“建议配置”写成实测结论。
- 不在 Git 记录真实姓名、电话、账号、令牌、URL、连接串、生产导出或漏洞细节；采用角色占位符及受控材料编号。
- 不将 `WAIVED` 作为默认状态；必须写明批准角色、风险、到期日、补齐计划、监控和回退条件。
- 不修改现有结论文本以“美化”状态；冲突以最新审查/报告为准，并保留原始引用。

### 4.2 数据结构

所有结构均为 Markdown 表格；不引入数据库或序列化格式。字段必须完整，未知值填 `TBD` 或 `BLOCKED`，不得留空制造歧义。

#### 4.2.1 证据记录

| 字段 | 说明 |
|---|---|
| `Evidence ID` | 如 `E-FR-301-01`，稳定且唯一 |
| `Requirement/Gate ID` | FR/NFR/Gate/Risk ID |
| `Evidence class` | `DEV_TEST` / `DEV_REVIEW` / `PROD_EQ_TEST` / `PROD_RECORD` / `THIRD_PARTY` / `EXTERNAL_DEPENDENCY` |
| `Environment` | `DEV` / `PROD_EQ` / `PROD` / `N/A` |
| `Source` | 仓库相对路径、commit 或受控材料编号；不填密钥/内部 URL |
| `Date/collector role` | 日期和角色占位；无证据填 `TBD` |
| `Integrity/summary` | 结果摘要、哈希/受控位置（如适用） |

#### 4.2.2 门禁条目

| 字段 | 说明 |
|---|---|
| `Gate ID` | 例如 `G-FUNC-01`、`G-PERF-03` |
| `Source requirement` | FR/NFR/流程条款 |
| `Rule/threshold` | 原始验收阈值或明确的通过条件 |
| `Minimum evidence` | 可使状态为 PASS 的证据类别与环境 |
| `Current evidence` | Evidence ID/路径，或 `None` |
| `Status` | 只能是 `PASS` / `FAIL` / `BLOCKED` / `WAIVED` |
| `Blocker/severity` | 上线阻断、限期豁免或提示 |
| `Owner/approver role` | 角色，不记录真实个人数据 |
| `Next action/due` | 可执行动作和期限/`TBD` |

#### 4.2.3 差距与豁免条目

| 字段 | 说明 |
|---|---|
| `Risk ID` | 如 `RISK-S01-01` |
| `Classification` | `VERIFY_PENDING` / `FUNCTION_GAP` / `EXTERNAL_DEPENDENCY` / `TECHNICAL_DEBT` |
| `Impact` | 影响的 FR/NFR、数据或发布面 |
| `Release disposition` | `BLOCK` / `MAY_WAIVE` / `NON_BLOCKING` |
| `Required closure evidence` | 需要的测试、配置核查、接口规范、第三方报告或实现任务 |
| `Waiver fields` | 风险、影响、批准角色、到期日、补齐任务、监控、回退条件；不完整不得为 WAIVED |

### 4.3 接口设计

不新增应用 API、数据库接口、消息主题或生产服务调用。P3 仅定义如下人工/受控证据接口：

| 接口 | 输入 | 输出 | 约束 |
|---|---|---|---|
| 证据登记 | 版本化文件、命令输出、受控附件编号 | Evidence ID 和矩阵引用 | 不上传原始生产敏感材料到 Git |
| 补测回填 | 获授权环境的原始结果、环境信息 | PASS/FAIL/BLOCKED 结论和证据引用 | 仅有执行责任角色可提交，审批者独立复核 |
| 风险豁免 | 差距条目 + 6 个必填豁免字段 | WAIVED 或拒绝 | 不允许口头/无期限豁免 |
| 发布决策 | 已核对的门禁矩阵 | 准入/不准入结论 | 任何 FAIL 或未批准 BLOCKED => 不准入 |

| 场景 | 触发条件 | 预期处理 |
|---|---|---|
| 证据路径失效 | 引用文件不存在或 commit 无法定位 | 标为 BLOCKED；修复引用或补充可复核证据，不得保留 PASS |
| 开发证据被误当生产证据 | DEV 测试/Mock/脚本语法检查被标为 PROD_EQ PASS | 降级为 DEV 证据；补测项恢复 BLOCKED |
| 报告冲突 | P0-P2 不同报告描述不同状态 | 以最新审查结论和原始输出为准；保留冲突说明并升级给负责人 |
| 没有目标环境/窗口/授权 | 无 K8s、DB、同城双活或变更批准 | 不执行；门禁保持 BLOCKED，记录具体前置缺失 |
| 性能/演练未达标 | 指标超过阈值、错误率超限、数据校验失败 | 标为 FAIL，停止扩大测试，执行既有回退/清理流程，生成整改任务 |
| 演练破坏业务或数据 | 故障注入/恢复异常 | 立即按授权 Runbook 回退，保留事件记录；不得继续其他注入 |
| 发现敏感信息 | 文档、日志、diff 含密钥/令牌/个人信息/连接串 | 停止提交，移除并使用受控引用；按安全流程处置已暴露信息 |
| 外部规范缺失 | 财务/监管/IAM/SSO 接口文档未提供 | 归为 EXTERNAL_DEPENDENCY，不伪造联调结论 |
| 试图口头豁免 | 无书面审批/到期日/补齐任务 | 状态不得 WAIVED，仍为 BLOCKED |
| 回退未验证 | 无回退负责人、触发条件或耗时证据 | 发布门禁 BLOCKED；补回退演练卡 |

## 6. 测试策略

P3 是文档/证据任务，测试重点是可追溯性、结构完整性与诚实性，不运行生产级功能或破坏性测试。

| 测试类型 | 测试内容 | 验收标准 |
|---|---|---|
| 覆盖性核对 | 46 FR、24 个唯一 NFR/27 条原子门禁、流程 §9 门禁是否均映射 | 数量与范围匹配；每项有唯一状态和下一步；NFR-E01~E03 有独立 PV-ARCH 卡 |
| 路径/链接核对 | P3 文档引用的仓库相对路径、脚本和报告是否存在 | 100% 可定位；不存在的外部材料显式 `CONTROLLED-LOCATION-TBD`，不伪造链接 |
| 状态机核对 | 枚举是否仅为 PASS/FAIL/BLOCKED/WAIVED；PASS 证据环境是否匹配 | 无无证 PASS；WAIVED 六字段完整 |
| 真实性核对 | 与 P2-02~P2-05 报告交叉验证 | 未实测项保持 BLOCKED；功能缺口不误列为待测 |
| 敏感信息检查 | 文档、diff、示例命令是否含真实凭证/连接串/数据 | 只含占位符，不含敏感信息 |
| 范围核对 | `git diff --check`、`git diff --name-only` | 仅允许需求/计划/任务与 `delivery/release-readiness/` 文档；无业务代码、迁移、生产配置变动 |
| 可执行性走查 | 选择每类补测至少一项，检查是否包含前置、步骤、指标、停止/回退、结果归档 | 负责人无需隐含知识即可发起授权申请与执行 |

无需因仅有文档改动强制运行完整 Maven/前端回归；不得声称它们在 P3 中被重新执行。若 Codex 执行文档内已有安全的本地静态检查，只能记录实际命令及结果。

## 8. 验收标准

1. **范围**：P3 仅包含获允许的文档；`git diff --name-only` 无业务代码、迁移、生产配置、密钥、脚本或历史报告篡改。
2. **FR 追溯**：46 FR 均有可复核条目，包含状态、开发/生产等价环境分层、证据/阻断原因、责任角色和下一步；Mock/外部依赖不被误标为 PASS。
3. **门禁覆盖**：覆盖流程 §9 的功能、数据库、持久化、安全、测试、性能、合规、运维门禁，并映射原 24 个唯一 NFR：NFR-P01~P07、A01~A05、S01~S03、E01~E03、C01~C03、M01/U01/G01；按复合阈值拆分后共 27 条原子门禁。
4. **状态真实性**：仅使用 PASS/FAIL/BLOCKED/WAIVED；每个 PASS 有目标环境匹配的最低原始证据；每个 WAIVED 有风险、范围、批准角色、到期日、补齐任务、监控和回退条件。
5. **补测可执行**：所有未实测生产级项目有前置、环境、数据、步骤、指标、通过/失败、停止、回退/清理、原始输出归档字段；无授权前状态为 BLOCKED。
6. **差距清晰**：MFA/IAM/SSO/字段 ABAC/证书认证、真实财务/采购/监管对接、后端 SCA、DAST、TLS、等保、三年审计留存、达梦/OceanBase 等缺口按正确类别记录，且有上线处置结论。
7. **准入规则**：清单明确“任何 FAIL 或未批准阻断 BLOCKED 即不准入”；不得产生正式生产上线批准。
8. **安全与质量**：所有相对路径可定位，`git diff --check` 通过，不含真实敏感信息，不虚构任何环境实测结论。

## 9. 风险与回滚

| 风险 | 等级 | 控制措施 | 回滚方式 |
|---|---|---|---|
| 误将开发验证表述为生产验收 | 高 | 证据层级 + 状态机 + PASS 最低证据规则；对照 P2 原始报告 | 更正状态为 BLOCKED，保留更正记录，禁止据此发布 |
| 隐藏功能/外部依赖缺口 | 高 | 差距台账区分 VERIFY_PENDING/FUNCTION_GAP/EXTERNAL_DEPENDENCY | 恢复阻断状态，创建独立实现/联调任务 |
| 文档泄露生产敏感信息 | 高 | 占位符/受控引用、关键词扫描、人工审查 | 在提交前移除；如已外泄按安全事件处置与密钥轮换 |
| 门禁矩阵遗漏原始要求 | 中 | FR/NFR/流程 §9 覆盖核对和数目检查 | 补条目后重审，不签发准入 |
| P3 文档漂移或重复既有报告 | 中 | 只链接事实源，P3 不改写历史结论 | 删除重复/错误材料，保持单一事实源 |
| 后续人员把模板当成批准 | 高 | README 与准入清单显著声明“当前为不准入准备包，非正式批准” | 撤回/修订模板并通知发布负责人 |
| 因文档任务超范围改动系统 | 中 | 严格路径白名单和 diff 核对 | `git restore` 无关改动；重新审查 |

**文档回滚**：P3 仅新增/追加 Markdown，回滚可通过删除 `delivery/release-readiness/`、阶段专用任务文档，或还原 `docs/requirements.md` 与 `tasks/requirement-analysis.md` 的 P3 附录完成；不涉及数据库、服务或生产配置回滚。
