## 1. P1 定位与目标

P1 目标是让功能更完整、更接近招采验收口径。P0 已完成上线阻断修复（落库、事实源、审计防篡改、真实依赖 E2E），P1 在此基础上补齐验收增强项：Connector 合约化、目录治理、质量报告、监管报表、财务适配。

**P1 与 P0 区别**：P0 是"能上线"（阻断修复），P1 是"能验收"（功能完整、口径贴近招采）。

## 2. 第一性原理分析

### 2.1 用户真正要解决什么？
P0 让系统跑通了主链路，但验收方（招采）需要看到：接入是规范化的（Connector 合约）、数据资产可追溯（目录治理）、质量可量化（质量报告）、合规可报送（监管报表）、计费可对接（财务适配）。这些是验收口径的硬指标。

### 2.2 最小可行结果
5 个验收增强项各自闭环，每项有可演示的产出 + 测试 + 证据。

### 2.3 任务依赖与执行顺序

| 编号 | 主题 | 依赖 | 涉及模块 | 可并行性 |
|---|---|---|---|---|
| P1-01 | Connector 合约 | P0-03（已合入） | platform-pipeline.ingest | 独立，可先做 |
| P1-02 | 目录治理 | P0-07（已合入） | platform-pipeline.catalog | 独立，可并行 |
| P1-03 | 质量报告 | P0-03（已合入） | platform-quality | 独立，可并行 |
| P1-04 | 监管报表 | P0-05（已合入） | platform-billing.stats | 独立，可并行 |
| P1-05 | 财务适配 | P0-06（已合入） | platform-billing | 独立，可并行 |

**关键判断**：5 个任务依赖均已满足，且模块互不交叉（ingest/catalog/quality/billing.stats/billing），**理论可全部并行**。但考虑审查带宽与质量，建议串行执行（每任务：任务单→实现→审查→合并→下一任务），与 P0 节奏一致。

### 2.4 建议执行顺序

按"验收价值优先 + 复杂度递增"排序：

1. **P1-01 Connector 合约**（ingest）：接入规范化是验收基础，且现有 8 适配器已就绪，扩展接口+contract test，复杂度中。
2. **P1-02 目录治理**（catalog）：血缘/质量摘要/使用统计，P0-07 已铺目录申请，自然延伸。
3. **P1-03 质量报告**（quality）：报告持久化+导出，P0-03 落库基础已就绪。
4. **P1-04 监管报表**（billing.stats）：模板/生成/脱敏/回执，依赖 P0-05 事实源，复杂度较高（报送回执）。
5. **P1-05 财务适配**（billing）：Finance/Purchase adapter，mock 对接，复杂度中（接口规范待采购方，本任务做框架+mock）。

## 3. P1 各任务概要

### P1-01 Connector 合约
- **问题**：现有 ProtocolAdapter 接口仅 protocol()+fetch()，无能力声明（支持增量/全量/断点续传/格式等），验收无法证明接入规范化。
- **产出**：SourceConnector 规范（扩展接口声明能力：协议/格式/同步模式/断点续传/限流）+ 能力矩阵（8 适配器 × 能力）+ 每类协议 contract test。
- **验收**：每类协议 contract test 通过。

### P1-02 目录治理
- **问题**：P0-07 目录申请已闭环，但目录详情无血缘/质量摘要/使用统计，验收无法追溯资产。
- **产出**：目录血缘（资产→服务/接入任务关联）、质量摘要（按资产聚合质量分）、使用统计（调用量/申请量）。新表 + 目录详情端点扩展。
- **验收**：目录详情可追溯（血缘+质量+使用）。

### P1-03 质量报告
- **问题**：质量检查有结果但报告不持久化、不可导出，验收需按维度生成报告。
- **产出**：质量报告持久化（t_quality_report）+ 按合作方/资产/服务生成 + 导出（Blob/文件）。
- **验收**：按合作方/资产/服务生成报告并可导出。

### P1-04 监管报表
- **问题**：统计有 dashboard/audit，但无监管报表模板/生成/脱敏/回执，验收需报文和回执入库。
- **产出**：监管报表模板配置 + 生成（合规/来源/个人信息）+ 脱敏 + 报送回执入库。注：监管标准待明确（requirements Q-05），本任务做框架 + mock 报送。
- **验收**：报文和回执入库。

### P1-05 财务适配
- **问题**：计费有账单生成，但无财务/采购系统对接，验收需对接框架。
- **产出**：FinanceAdapter/PurchaseAdapter + mock 成功/失败/重试。注：接口规范待采购方（requirements Q-06），本任务做适配框架 + mock。
- **验收**：mock 成功/失败/重试通过。

## 4. P1 共性约束

- 每任务遵循 P0 工作流：`tasks/codex-task-P1-xx.md`（基于第一性原理计划）→ Codex 实现 → Claude 审查 → 返工/通过 → 合并 master。
- 分支命名：`ai/p1-<主题>`（如 `ai/p1-connector-contract`）。
- 版本号：续 V016，P1 各任务按需用 V017+（避免冲突，实际以任务单为准）。
- 国产化：新表/SQL 须 MySQL + 达梦双库（P0-02 策略），通过 MigrationDialectCompatibilityTest 守护。
- 不破坏 P0 既有闭环（落库/事实源/审计/真实依赖）。
- 上线前门禁清单持续维护（P0-08 DB REVOKE、P0-10 V015 checksum、P0-10 E2E 证据）。

## 5. P1 风险

| 风险 | 控制 |
|---|---|
| P1-04 监管报送标准未明确（Q-05） | 本任务做框架+mock，标注"真实报送待标准明确"，不阻断验收框架 |
| P1-05 财务接口规范未明确（Q-06） | 同上，adapter 框架+mock，真实对接待采购方规范 |

| P1 改动破坏 P0 闭环 | 每任务全量回归 mvn test + 前端测试 |

- 5 个任务各自闭环：任务单→实现→审查→合并 master。
- 每任务有可演示产出 + 测试 + 证据。
